## 3. Secret Files (NOT in GitHub — copy manually)

These live only on your local machine. Never commit them.

### `.env`
```
EXPO_PUBLIC_SUPABASE_URL=https://YOUR-PROJECT.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here
```
Get from: Supabase dashboard → Settings → API

### `.env.mock`
```
EXPO_PUBLIC_USE_MOCK=true
EXPO_PUBLIC_SUPABASE_URL=http://localhost
EXPO_PUBLIC_SUPABASE_ANON_KEY=mock
```
(Static — copy as-is for any project using MockAuthContext)

### `.secrets`
```
SUPABASE_MANAGEMENT_TOKEN=sbp_xxxx
NOTION_TOKEN=ntn_xxxx
RESEND_API_KEY=re_xxxx
```
Get from: Supabase dashboard → Account → Access tokens; Notion → Settings → Integrations

## 5. Backend Setup Steps

### Supabase
1. Create project in eu-west-2 (or your preferred region)
2. Copy Project URL + anon key → `.env`
3. Open SQL Editor → paste and run `supabase/schema.sql`
4. That creates all 8 tables, RLS policies, and RPC functions
5. Enable Email Auth in Supabase → Auth → Providers
6. Set invite email template (or use `supabase/functions/send-invite-email`)
7. Add `SUPABASE_MANAGEMENT_TOKEN` to `.secrets`

### Notion
1. Create workspace
2. Create integration at notion.so/my-integrations
3. Add token to `.secrets` as `NOTION_TOKEN`
4. Run `scripts/notion_setup.py` to rebuild the workspace structure


| `app.json` | `name`, `slug`, `ios.bundleIdentifier`, `android.package` |
| `constants/colors.ts` | Brand colours |
| `constants/challenges.ts` | Domain-specific templates |
| `constants/icons.ts` | Domain-specific icons |
| `supabase/schema.sql` | Tables for your domain |
| `.env` | New Supabase URL + anon key |
| `.secrets` | New management tokens |
| `.vercel/project.json` | New Vercel project ID (from `vercel link`) |

## 8. Key Architecture Rules (don't break these)

- Gem mutations via RPC only — never `supabase.from('family_members').update({ gem_balance: x })`
- RLS is the security layer — never bypass it with service_role on the frontend
- `EXPO_PUBLIC_` prefix is mandatory for all env vars used in the app bundle
- Icon DB values are MaterialIcons names — always render via `<ItemIcon>`, not `<MaterialIcons>` directly
- Both MaterialIcons and MaterialCommunityIcons fonts must be preloaded in `_layout.tsx` for web export
